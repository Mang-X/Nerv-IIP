# AI 边界与治理说明

本文档定义 Nerv-IIP 平台中的 AI 能力边界、工具分类、模型接入方式和治理原则。

## 总体定位

平台的 AI 能力服务于三类场景：

1. 平台运维与管理查询。
2. 平台受管动作的安全执行与审批。
3. 业务文档与运行知识的检索增强。

平台不把自己定位为模型托管平台，而是定位为模型调用治理者、工具暴露治理者与知识服务消费者。

## 核心边界

### AI Integration

- 负责模型提供方配置。
- 负责 MCP Server、Skill 注册、工具授权与执行审批。
- 工具授权必须消费 IAM 的身份、组织、环境、权限范围和外部客户端授权事实。
- 负责工具调用审计和人工确认挂点。
- 不负责编排知识索引，不负责模型托管。

### Knowledge

- 负责知识源接入、文档处理、分块、嵌入、索引、检索、权限过滤和引用回显。
- 对 AI Integration 和平台服务暴露统一检索能力。
- 不负责工具治理与执行审批。

### Ops 与 AppHub

- Ops 仍然拥有动作任务与审计闭环。
- AppHub 仍然拥有应用与实例事实。
- AI Integration 调用这些服务的能力时，只是治理工具访问，不接管它们的领域事实。

## 模型接入原则

1. 平台不提供内置模型托管。
2. 模型调用统一通过外部 API 配置或外部 Agent 客户端接入。
3. 平台内部凡是需要直接调用模型的场景，默认通过 Microsoft.Extensions.AI 抽象接入。
4. 平台不把具体模型 SDK 散落到各个服务中。

## 工具分类

### 只读查询类

- 应用状态查询
- 实例详情查询
- 任务记录查询
- 日志检索
- 知识检索

首批优先开放只读查询类工具。

### 低风险执行类

- 重启
- 重新拉取状态
- 触发标准备份

这类工具必须有授权与审计。

### 高风险执行类

- 破坏性运维动作
- 跨环境批量变更
- 影响不可逆的数据操作

这类动作必须预留审批、人工确认或双重确认机制。

## 安全与治理规则

1. 所有工具调用都应记录审计信息。
2. 所有跨环境操作都必须显式携带组织与环境上下文。
3. 工具权限模型不能绕过 IAM 的身份、RBAC、组织环境边界和外部授权事实。
4. AI 不得成为运维事实的真相源，只能通过平台已有服务查询或触发受管动作。

## 官方 .NET AI 生态的使用策略

1. Microsoft.Extensions.AI：默认统一 AI 调用抽象。
2. Microsoft.Extensions.DataIngestion：默认文档引入主路径。
3. Microsoft.Extensions.VectorData：默认向量索引抽象。
4. Microsoft Agent Framework：仅在确有长流程、多代理或复杂人机协同场景时评估引入。

## 当前冻结结论

1. 平台首期不做模型托管。
2. 平台首期优先交付 MCP 查询工具与低风险动作。
3. 知识能力与工具治理分属两个服务边界。
4. 任何执行类能力都必须带权限与审计。
