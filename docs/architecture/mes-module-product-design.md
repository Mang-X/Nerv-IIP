## 1. 域定位（ISA-95）

MES（ISA-95 Level-3）分**四大运营域**：生产运营、质量运营、维护运营、库存运营；每域走**同一通用活动模式**：
`定义 → 资源管理 → 详细排程 → 派工 → 执行 → 数据采集 → 跟踪 → 分析`。

**本平台「制造执行」模块 ≈ 生产运营管理（Production Operations Management）。** 质量 / 设备 / 库存 / 需求计划是**独立域**（已有独立模块）。

**推论（硬约束）**：
- MES 模块的 IA 应**按生产运营活动流编排**，不按"管理动作"凑组。
- 质量、设备、库存的对象**不当 MES 主对象**；MES 内只保留它们的**只读窗口 / 跳链**，主维护在各自模块。

## 2. 目标 IA / 导航重构

### 2.1 现状问题（三视角共识）
1. **「异常与协同」是反模式杂物抽屉**：把 产能(分析) / 规则排程(排程) / 班次交接(执行协同) / 质量·设备窗口(他域) 五个**弱关系实体硬凑**一组。
2. **同一 ISA-95 活动被物理打散**：生产计划 / 规则排程（都属排程）分居两组；在制跟踪 / 追溯（都属跟踪）分散。
3. **他域窗口被当 MES 主菜单项**：质量与不良、设备与停机占用 MES 顶层菜单，与独立「质量管理」「设备监控」模块认知冲突。
4. **同对象多视图被当独立页**：派工看板（工单的看板视图）、生产准备检查（开工 readiness 视图）各占一个菜单项。

### 2.0 命名原则（可发现性，硬约束）
> **导航 / 页面 / 按钮一律用一线操作动词**（领料 / 报工 / 派工 / 开工 / 完工 / 交接），让业务人员**一眼找到操作点、不靠翻文档**。
> ISA-95 的功能术语（执行准备 / 数据采集 / detailed scheduling）**只用于本设计文档的结构分组，不进界面**。**结构对齐 ISA-95，标签对齐一线话。**
> 反例（本文档 v1 踩坑）：用「执行准备」「报工采集」当组名 → 一线找不到「领料」；用「齐套与物料」当页名 → 藏掉了每天高频的「领料」操作。

### 2.2 目标导航树（结构按 ISA-95 活动流，标签用一线话）
```
制造执行（MES = 生产运营管理）          ← 括号内为 ISA-95 活动，仅说明，不上界面
├─ 生产总览        /mes                  [域首页 / Andon 总览]          (Analysis 总览)
├─ 计划与排程      生产计划 /mes/plans · 排程 /mes/schedules            (Scheduling)
├─ 工单与派工      /mes/work-orders[+详情]（派工看板收为页内 Tab）       (Dispatching)
├─ 领料与齐套      /mes/materials（开工准备 foundation 收为 Tab）         (Execution Readiness)
│                  ← 由「齐套与物料」改名，**领料**进标题；主操作=发起领料/领料申请
├─ 工序执行        工序执行 /mes/operation-tasks · 班次交接 /mes/handovers (Execution)
├─ 报工与完工      报工记录 /mes/production-reports · 完工入库 /mes/receipts (Data Collection)
├─ 在制与追溯      在制跟踪 /mes/wip · 追溯查询 /mes/traceability          (Tracking)
└─ 生产分析        异常与产能 /mes/capacity                              (Analysis)
★ 移出 MES 顶层菜单（改他域跳链 / 详情内联只读）：质量与不良 → 质量管理；设备与停机 → 设备监控
```
菜单项从 ~16 压到 ~11；每组 1-2 页、语义单一、顺序即生产流；**标签即一线操作**。

### 2.3 合并 / 拆分 / 迁移清单
| 操作 | 对象 | 目标 | 理由 |
|---|---|---|---|
| 迁移 | 规则排程 | 异常组 → 计划与排程 | 属 detailed scheduling |
| 迁移 | 班次交接 | 异常组 → 班中执行 | 执行协同非异常 |
| 拆独立 | 异常与产能 | 异常组 → 生产分析 | analysis 活动 |
| 收 Tab | 派工看板 → 工单页视图 | 同对象不同视图 → 页内 Tab |
| 收 Tab | 生产准备检查 → 齐套页 Tab | 开工 readiness 平级视图 |
| 移出 | 质量与不良 / 设备与停机 | 他域跳链 + MES 详情内联只读 | 他域主对象不占 MES 菜单 |
| 解散 | 「异常与协同」分组 | 五实体各归其位 | 杂物抽屉反模式 |

> **唯一需产品确认**：生产计划 /plans 与 规则排程 /schedules 的边界——是「计划数据 vs 排程规则配置」（→ 一页双 Tab）还是两套并行引擎（→ 同组两页）？

## 3. 核心 UX 流修复（执行价值流）

三条核心旅程 + 断点（操作员旅程断点最严重）：

### 3.1 报工就地化（最高优先）
**问题**：工序页「报工」是 `openRoute('/mes/work-orders')` **整页跳工单列表 + query + watch 弹窗** —— 全模块最严重的上下文丢失点；报工弹窗只读工单/工序号，其余纯手填。
**方案**：把报工表单抽成**共享抽屉组件** `ProductionReportDrawer`，由 工单行 / 工序行 / 在制行 **就地调起**（删除 query→弹窗 hack）。工序进入锁定该工序；工单进入默认选 Running/Ready 工序（非 `operationTasks[0]`）。

### 3.2 动作接力（消灭死胡同）
派工指派完、急单提交完、报工完工后都缺「下一步」。每个关键动作成功后给**带参后续动作**：
- 派工成功 → 「查看该工序执行」
- 急单返回的受影响工单 → 每个 workOrderId 做成**跳详情链接**（现为纯文本）
- 报工勾「完成当前工序」且**末道工序** → toast 给「去完工入库」带 workOrderId 直达预填

### 3.3 阻塞项可点
派工看板 / 驾驶舱「展示阻塞原因文字但点不动」。阻塞原因应是**带 workOrderId 的跳转**，一键到处理页（齐套/质量/设备），闭合 ISA-95「执行→排除阻塞」环。

### 3.4 角色化落到列表过滤
驾驶舱 roleLanes 已有骨架，但「点进去=全量列表」。给 `operation-tasks` 加**「我的工序」快捷过滤**（`assignedUserId` = 当前登录用户），操作员不再在所有人的工序里翻找。

### 3.5 一线少填（报工自动带出）
照 `dispatch` 派工弹窗的模式：身份/工序/设备/班次**系统带出**，一线只补 良品数 / 报废数 / 是否完成三项。良品数默认值 = 计划数 − 已报数（剩余应报）。

### 3.6 异常/排程双向链
异常三页（capacity/downtime/quality）与规则排程是孤岛 → 补「源工单回链」（异常行带 workOrderId+operationTaskId）与「工单→排程」正向链。
**统一带参契约**：跨工序操作一律携带 `workOrderId + operationTaskId + workCenterId + skuId`。

### 3.7 领料闭环可视化（跨域：MES 领料申请 → WMS 出库）
**事实**：`MaterialIssueRequestRow` 已含 `wmsRequestId`（连 WMS 出库）+ `requestedQuantity`/`receivedQuantity`（收料进度）+ `status` —— **后端闭环已建**。但前端「齐套与物料」页不暴露这条链，用户发起领料后看不到后续 → 体验上"断"。
**方案**：领料申请行显式做成「**已收 / 应领 进度 + 状态徽章**」，并把 `wmsRequestId` 做成**跳 WMS 出库**的链接（不显 GUID，显「查看出库」）。让「工单缺料 → 发起领料 → WMS 出库 → 收料回写 → 齐套」整条在 MES 侧一眼可见、可追。**领料是开工前的物料门，必须可发现、可追踪。**

## 4. 改造批次与优先级（#420 感知）

价值/频次：越靠「执行+报工采集」越高频、卡住越致命；越靠「分析/配置」越低频。

| 批 | 内容 | 理由 |
|---|---|---|
| **批 1 执行采集主干** | 工序执行 → 报工记录 → 齐套与物料（含 §3.1 共享报工抽屉、§3.4 我的工序、§3.5 少填） | 最高频价值地基，且**不被 #420 卡**，投入产出比最高 |
| **批 2 入口与导航** | §2 IA 导航重构 + 生产驾驶舱（KPI 含义化、§3.2/§3.3 接力与可点阻塞）+ 在制跟踪 | 把执行页串成「角色→任务→报工」动线 |
| **批 3 派工链（壳先行）** | 工单与派工 + 派工看板 | 价值最高但**被 #420 卡内核**（ID→名称、状态 Select）；先做布局/动作/角色泳道，内核挂 #420 不空等 |
| **批 4 班次闭环** | 班次交接 → 生产准备检查 → 完工入库 | 补全「一天的开始与结束」，流程固定风险低 |
| **批 5 排产/分析/他域链** | 规则排程 → 生产计划 → 异常与产能 → 追溯；质量/设备仅做跳独立模块的轻链接 | 低频或有独立模块兜底 |

**MVP（车间能跑）**：工单与派工、派工看板、工序执行、报工记录、齐套与物料、生产驾驶舱（+ 配套 PDA 领料/工序/报工/收货）。

**风险**：派工/工序核心交互（派工 Dialog、可开工判断 `describeMesReadinessReason`）是生产关键路径，**只补 IA/层级/空态/接力，不重写核心逻辑**——避免为改而改、改坏停产。

## 5. 复用基础数据经验（playbook 落地到 MES）
- **FE-2 区块拼装**、不裸表、不改 shadcn 原版（同主数据 §0）。
- **SectionCards 非默认**：MES 驾驶舱/各页**禁机械计数 KPI**（`本页X` / 裸 `reduce` 计数）→ 改**含义化语义**（待开工 / 超期 / 阻塞待排除，能驱动动作）。
- **说人话**：UI 不暴露 GUID/工程语言。MES 的 ID→名称受 **#420** 卡 → facade 就绪前**标注**，不硬显 GUID 当标识。
- **IA 判定规则**（§6 of templates）：父子→树；同对象视图→页内 Tab(≤5、不进菜单)；弱关系→拆独立页；主+受控子集→主从。MES 按此判（§2.3）。
- **校验 on submit、toast 反馈、诚实数据、空态给出路**（同主数据 §5/§7.2）。
- **§1.5 两轮自检**（实现前 A + 完成后 B）每页过。
- **视觉验证不跳过**：每页改完用 Claude-in-Chrome 截真实运行页核对（本轮已打通）。

## 6. 行业标准依据与来源
- ISA-95：MES = 驱动制造运营有效执行的动态信息系统，管理「从订单下达到成品入库」；Level-3 分四运营域，每域同一活动模式（排程/派工/执行/采集/跟踪/分析/定义/资源管理）。
- 制造 UX：强对比、告警优先、减杂；清晰层级、路径短而可预测、实时反馈；看板可一眼读、状态集有限（Andon）；一线少填、按角色组织任务。
