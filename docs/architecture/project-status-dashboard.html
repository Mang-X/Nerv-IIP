下一阶段并行路线

#142：对象存储补强

  • 在 FileStorage metadata/local+tus MVP 之后，补 MinIO/S3 multipart provider 与大文件直传能力。
  • 保持业务服务只表达 fileId/FileReference，不直接管理对象存储、bucket 或凭据。
  • 验收需要覆盖 provider 切换、multipart 生命周期、checksum/size 与失败清理。

PDA / Mobile 业务入口

  • 围绕扫码、离线、WMS/MES/Quality 现场任务拆分移动 facade、Capacitor/scanner 和 UI issue。
  • 业务控制台继续走 BusinessGateway/BusinessConsole；移动端不直接跨服务拼装领域逻辑。
  • 附件、照片和证据仍统一走 FileStorage，不直连对象存储。

事件可靠性 hardening

  • Notification、AppHub、BusinessMES 和 BusinessMaintenance 已具备 PostgreSQL persistent DLQ 基线。
  • 继续补跨进程多服务 CAP 联调、持久 inbox、DLQ 自动 replay executor 和管理入口。
  • #77 P0 验收脚本继续作为业务链路回归入口。

发布 / 可靠性硬化

  • 生产 artifacts 已有基线；下一步补 Windows Service/systemd、备份恢复演练、seed 清单和现场诊断。
  • 真实 PostgreSQL/RabbitMQ/外部设备联调作为 hardening profile 增量，不阻塞 #77 P0 关闭。
  • 新脚本必须 dot-source scripts/lib/ScriptAutomation.ps1 并过治理检查。

已知风险

真实基础设施联调未默认开启

#77 P0 默认使用 TestServer 与本地验证 profile；PostgreSQL/RabbitMQ/外部设备联调仍需后续 hardening profile 覆盖。

对象存储仍是 post-MVP

#142 未完成前，FileStorage 不应被宣传为已具备生产级 MinIO/S3 multipart 直传能力。

客户交付闭环未完成

Compose、生产环境模板和 zip package 已有基线，但最终客户安装器、服务注册、备份恢复和现场运维手册仍需专项收口。
