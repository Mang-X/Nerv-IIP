下一阶段并行路线

业务 MVP 主线：#77

  • Worker A：验收 harness、fixture、event recorder、verify-business-full-chain-acceptance.ps1。
  • Worker B：工程到制造、计划到采购/生产链路。
  • Worker C：采购到库存到应付、订单到交付到应收、生产到成本链路。
  • Worker D：设备到维护到产能、WMS-WCS adapter 链路。

基础能力并行

  • FileStorage：Gateway/Console facade、FileUpload wrapper、tus 硬化。
  • Design System：业务页面所需组件先进入 @nerv-iip/ui。
  • Notification：修复组织/环境 headers，再做偏好、订阅、限流与 provider。
  • IAM / Ops / Observability：生产暴露策略、高风险动作审批、诊断附件与日志入口。

业务控制台切入点

  • 先做信息架构与模块导航，不急着把所有业务页一次铺开。
  • 第一屏建议选 MasterData + Inventory + Quality + MES 的列表/详情/创建流。
  • 附件、证据、照片、CAD 包统一走 FileStorage 与 FileUpload，不直连对象存储。

治理与验证

  • 每条业务链只通过公开 HTTP API 与 IntegrationEvent 可见事实断言。
  • 不跨 schema FK，不引用其它服务 Domain / Infrastructure。
  • 新脚本必须 dot-source scripts/lib/ScriptAutomation.ps1 并过治理检查。

已知风险

验收断言可能缺查询面

MES、WMS、Maintenance 等服务可能需要补少量只读 API，才能让 #77 不靠数据库直读。

Notification 页面上下文缺口

worker 核到前端调用 Notification facade 时未显式传 X-Organization-Id / X-Environment-Id，运行时可能 400。

业务界面落后于后端

目前 Console 已有平台实例、Ops、IAM、Notifications，但业务服务还没有相应的业务控制台页面。
