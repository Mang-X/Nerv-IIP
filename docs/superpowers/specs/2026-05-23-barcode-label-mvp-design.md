# BarcodeLabel MVP Design

## Goal

Build BarcodeLabel as the business fact source for barcode rules, label templates, label print batches and scan records.

BarcodeLabel supports WMS, MES, Inventory and mobile/PDA workflows, but it does not own warehouse task state, inventory quantities or business document lifecycle.

## Current State

BarcodeLabel has no service directory. MasterData owns SKU/barcode policy inputs and FileStorage owns file/template attachments. Inventory owns stock quantities and movement facts.

## Owned Facts

BarcodeLabel owns:

1. BarcodeRule: barcode type, prefix, length, checksum rule, allowed source document types and status.
2. LabelTemplate: template metadata, template file reference, variable schema and status.
3. LabelPrintBatch: print request, target document reference, label values, requested quantity, status and idempotency key.
4. LabelPrintItem: generated label value and optional file reference.
5. ScanRecord: source device, scanned value, source workflow, target document reference, scan result and idempotency key.

BarcodeLabel does not own:

1. Inventory on-hand, available, reserved or frozen quantities.
2. WMS inbound/outbound/task state.
3. ERP purchase, sales or invoice state.
4. MES work order or operation state.
5. FileStorage object keys, binary bytes or download grants.
6. SKU master facts, UOM or barcode policy ownership.

## API Surface

| API | Purpose | Permission |
| --- | --- | --- |
| `POST /api/business/v1/barcodes/rules` | Create or update a barcode rule. | `business.barcodes.templates.manage` |
| `POST /api/business/v1/barcodes/templates` | Create or update a label template. | `business.barcodes.templates.manage` |
| `GET /api/business/v1/barcodes/templates` | List active templates. | `business.barcodes.templates.manage` |
| `POST /api/business/v1/barcodes/print-batches` | Create a label print batch and generated label items. | `business.barcodes.print` |
| `GET /api/business/v1/barcodes/print-batches/{printBatchId}` | Read print batch detail. | `business.barcodes.print` |
| `POST /api/business/v1/barcodes/scans` | Record a scan result. | `business.barcodes.scans.write` |
| `GET /api/business/v1/barcodes/scans` | List scan records by device, value or source document. | `business.barcodes.scans.write` |

## Rules

1. Print batch creation requires an idempotency key and source document reference.
2. Repeating the same print idempotency key with the same payload returns the existing batch.
3. Repeating the same print idempotency key with a different payload is rejected.
4. Scan record creation requires source device, scanned value, source workflow and idempotency key.
5. Repeating the same scan idempotency key with the same payload returns the existing scan record.
6. Scan records are append-only business facts and do not mutate WMS, MES or Inventory state.
7. Template file IDs are FileStorage references only; object keys and long-lived URLs are forbidden.
8. Barcode values must be deterministic from rule, source document and sequence inputs when generated by the service.

## Events

BarcodeLabel publishes ADR 0011 envelope events:

1. `barcode.LabelPrintBatchCreated`
2. `barcode.LabelPrintBatchCompleted`
3. `barcode.LabelScanned`
4. `barcode.ScanRejected`

Events carry public document references, source device, scanned value, template/rule IDs and idempotency/correlation IDs. They must not carry FileStorage object keys.

## Permissions

Initial permission codes:

1. `business.barcodes.templates.manage`
2. `business.barcodes.print`
3. `business.barcodes.scans.write`

## Persistence

Default schema: `barcode`.

Required tables:

1. `barcode_rules`
2. `label_templates`
3. `label_print_batches`
4. `label_print_items`
5. `scan_records`

Each table and business column requires schema comments. PostgreSQL migrations history must use `barcode.__EFMigrationsHistory`.

## Testing

Acceptance requires:

1. Domain tests for deterministic label generation, print idempotency and scan idempotency.
2. Domain tests rejecting blank scanned values, missing devices and conflicting idempotency payloads.
3. Web tests for route shape, authorization, validation and operation IDs.
4. Schema convention tests using `Nerv.IIP.Testing`.
5. Integration event converter/serialization tests for print and scan facts.
6. Tests proving no public response contains `objectKey` or `object_key`.
