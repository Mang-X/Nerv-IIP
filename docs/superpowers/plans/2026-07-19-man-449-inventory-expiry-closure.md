# MAN-449 Inventory Expiry Closure Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close every MAN-449/#803 acceptance gap with truthful Inventory data, server-side paging/counting, persisted expiry provenance, and backend-rule-driven Console actions.

**Architecture:** Inventory remains the source of truth. Availability and expiry-alert queries return the persisted batch dates/provenance plus explicit operation availability and block reasons; expiry alerts use database `CountAsync` plus `Skip`/`Take`. BusinessGateway mirrors the Inventory contracts, OpenAPI/codegen produces frontend types, and the Console consumes only generated fields.

**Tech Stack:** .NET 10, EF Core, FastEndpoints, xUnit, BusinessGateway, Hey API, Vue 3, Pinia Colada, Vitest, NvUI.

## Global Constraints

- Scope is MAN-449/GitHub #803 only; do not modify WMS inbound, Quality, ERP, DemandPlanning, Screen, or PDA pages.
- Do not hand-edit OpenAPI snapshots or generated API client files.
- Do not infer business operation state in the frontend from dates.
- The near-expiry default is 30 days and paging/counting must be server-side.
- Expiry thresholds remain centralized in `@nerv-iip/business-core`.
