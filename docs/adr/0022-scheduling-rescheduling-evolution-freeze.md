# ADR 0022: 排程演进冻结——规则源与引擎边界、三层重排模型与拆批后置

- Status: Accepted
- Date: 2026-07-23

## Context

ADR 0014 已冻结 APS lite 的 P0 边界：`BusinessScheduling` 拥有排程事实，`FiniteCapacityScheduler`（`aps-lite-v1`）作为纯计算内核消费标准化 `SchedulingProblem` 输出 `SchedulePlan`，MES 只消费方案不隐藏私有调度算法。此后排程域持续长出新能力：#904/MAN-383 失效标记与事件传播、#957/MAN-385 release 版本治理（supersede）、#923 排程锁与手工 override、#1053/MAN-584 统一订单紧急度 V1（`order-urgency-v1`）、#1050/MAN-581 APS Lite 100/500/1000 规模基准，以及 #1078/MAN-580 计划员工作台（`WorkingScheduleDraft` 闭环：批量生成、受治理 plan revision、锁保持、失效影响与 base-candidate 对比）。

在此基础上，四个演进方向已立项但尚未冻结边界：#763 规则/约束来源与排产引擎的分离、#1051/MAN-582 现场执行偏差驱动重排、#1058/MAN-588 自动局部重排引擎、#1052/MAN-583 订单拆批/转移批量/多设备并行。若不先冻结概念分层与依赖顺序，这些 issue 会以"逐票长出来"的方式相互穿插，重演 FileStorage v1 需要整体重构的教训。

本 ADR 冻结演进层的概念边界、交付顺序与护栏；不改变 ADR 0014 已冻结的 P0 边界，不在本 ADR 引入任何求解器实现。

## Decision

### A. 规则源与排产引擎分离（承 #763）

1. **规则源（RuleProvider/ConstraintProvider）与排产引擎（SchedulingEngine）是两个概念，永不合并。** 规则源产出约束、优先级、目标权重等输入事实；引擎消费标准化 `SchedulingProblem` 产出 `SchedulePlan`。规则引擎不做排产决策，排产引擎不解释业务规则来源。
2. Scheduling 服务层引入可替换抽象：`ISchedulingEngine`（默认实现仍为 `aps-lite-v1` 确定性有限产能启发式）与规则/约束 provider 族（现有硬编码策略先原样封装为默认 provider，不要求立即配置化）。抽象落位后，`FiniteCapacityScheduler` 的行为、排序规则（ADR 0014 §13）与输出契约不得改变。
3. **后续任何 solver/优化器（OR-Tools、商业求解器、启发式优化器）只能以 engine adapter 形式接入**，输入输出仍是稳定 `SchedulingProblem` → `SchedulePlan`；MES、BusinessGateway 与前端不感知计算方式。引入任何 solver 需要新 ADR，本 ADR 不授权。
4. **可追溯冻结为契约**：`SchedulePlan` 持久化与读面必须记录 `engineId/engineVersion`、生效的 rule/profile/version 摘要与约束来源摘要，支撑审计、解释与同输入回放。现有 `AlgorithmVersion` 字段语义并入 `engineVersion`，不另造第二套版本字段。
5. MES legacy `RuleScheduler` 冻结为 **deprecated**：不得新增任何依赖；MES 局部补排需求一律收敛为消费 `SchedulePlan` 或向 Scheduling 提交重排请求。移除时机由独立清理 issue 跟踪，不阻塞本 ADR 各项。

### B. 三层重排模型（承 #1051/#1058）

6. 重排能力冻结为三层，**每层建立在下层之上，禁止跨层实现**：
   - **L1 失效与影响标记**（已交付：#904 失效事件、#957 版本治理）：识别现场变化，使方案失效并传播事件；不生成新方案。
   - **L2 人工重排工作台**（已交付：#1078 `WorkingScheduleDraft`）：计划员在失效影响可见的前提下重新生成、锁保持、base-candidate 对比、受治理发布。L2 的"重新生成"允许全量重算。
   - **L3 自动局部重排引擎**（#1058，未交付）：影响分析 + 冻结策略 + 仅对受影响链路生成候选方案；单设备停机不得触发全厂重算。
7. **L3 的影响分析与候选生成是纯计算单元**，遵守 ADR 0014 §12 的全部约束（无 IO、无系统时间、无随机数、确定性、同输入同指纹）。其输入是 `SchedulingProblem` + 当前 `SchedulePlan` + 标准化的偏差事实集，输出是 `ReschedulingImpact`（受影响订单/工序/资源/依赖链、可冻结集、需重算集）与候选 `SchedulePlan`。
8. **冻结策略是显式契约**，默认保持：已完成工序、已开工工序、人工锁定工序（#923）、计划员确认的稳定窗口。冻结集内的 assignment 原样保留且仍参与有限产能校验（ADR 0014 §18 语义不变）。
9. **候选方案是 preview 态方案**，走 ADR 0014 §14 既有状态机；发布复用 #957 的 release/supersede 治理与 #1078 的工作台确认流，**不新造第二条发布通道，不做无人值守自动发布**。候选对比的 KPI 集冻结为：准交率变化、延期订单数、移动工序数、资源利用率变化、未排工序数、锁定保持情况——与 #1050 基准口径同源，每个移动的工序必须给出可解释原因码。
10. **#1051 偏差驱动重排是 L3 之上的流程层，不是第二个引擎**：来源域（MES 报工/进度、IndustrialTelemetry 设备状态、Quality 阻断、Inventory 齐套）继续拥有各自事实，Scheduling 只消费标准化偏差投影（复用 ADR 0014 §20–§22 的 #207 语义与既有 provider，不新增跨 schema 读取）。流程为：偏差事实 → L3 影响分析 → 候选生成 → 计划员确认 → 发布。#1051 可分段交付：影响分析与提醒先行，候选生成随 L3 落地。
11. 交付顺序冻结为：**A 组抽象（§1–§4）→ L3 引擎核心（§7–§9）→ #1051 流程层（§10）→ #1052 拆批**。每个实现 PR 必须引用本 ADR；违反分层（如在 MES 或前端实现影响分析、绕过版本治理直接发布候选）按架构回归处理。

### C. 拆批与并行排产后置（承 #1052）

12. 订单拆批/转移批量/多设备并行**显式后置到 L3 稳定之后**。冻结两条前置原则：拆批决策发生在计划层，产出带完整追溯关系（父订单 → 批次）的计划提案；MES 工单的实际拆分只能由 MES 命令按已发布方案落地，计划层不直接改写执行域。工序重叠/转移批量建模进入 `SchedulingProblem` 契约前需要契约版本演进评审。

### D. 输入事实与性能护栏

13. 紧急度输入统一走 `order-urgency-v1` 结果（#1053）：引擎与影响分析消费其统一等级/执行风险作为优先级输入，不得在排程域内出现第二套紧急度或健康度计算。设备健康/风险类新事实（#1055）接入执行风险维度前，必须先在紧急度模型侧登记为新贡献项并保持可解释分解，不得直接注入引擎。
14. **性能预算以 #1050 基准为基线**：stress 档（1000 单/4000 工序/24 资源）全量重算中位约 36 秒。L3 局部重排在受影响子集上的耗时必须显著优于对等全量重算，且基准脚本扩展局部重排档位（受影响子集规模 × 候选数）后纳入同一证据体系；无基准证据的"局部"实现不接受。
15. 新增 HTTP 端点全部遵守 facade coverage 两跳 DoD（`exposed`/`deferred`/`internal` 声明 + 门禁）；跨服务共享的新契约放 `Nerv.IIP.Contracts.Scheduling`，enum 维持 camel-case string（ADR 0014 §11/§16）。

## Rationale

1. #763 的核心洞察成立：把规则来源和排产引擎拆开，未来换引擎（solver）不动规则，改规则不动引擎；不拆则每次演进都是全域改动。
2. 三层重排模型把"动态重排"从一个大而模糊的目标拆成已交付（L1/L2）与待交付（L3）的清晰边界——领导演示已可用 L1+L2 讲"失效 → 重新生成 → 对比 → 发布"的真实故事，L3 补的是"只动受影响链路、保持全局稳定"的精确性，两者不混淆也不互相等待。
3. 候选方案复用 preview/release 既有状态机与版本治理，避免自动重排绕过审计与授权，也让 L3 的产出天然可被 #1078 工作台承接。
4. 拆批后置是因为它同时改 Problem 契约、引擎语义与 MES 落地命令，任何一层未稳定时引入都会造成三层同时返工。
5. 性能护栏防止"局部重排"退化为换了名字的全量重算——那会让 L3 失去存在意义。

## Consequences

1. #1058 实现范围收敛为本 ADR §7–§9（引擎核心 + 影响分析 + 候选生成 + KPI 对比），其 issue 中"输入事件"清单降级为 §10 流程层的偏差投影来源，不在引擎内直接消费集成事件。
2. #1051 实现范围收敛为 §10 流程层，依赖 §7 影响分析；可先交付影响分析与提醒段。
3. #1052 保持 backlog，启动前置条件为 L3 交付且基准扩展完成。
4. #763 由本 ADR 关闭决策部分；其代码落地（§2 抽象 + §4 追溯字段）拆独立实现 issue。
5. `docs/architecture/business-platform-domain-architecture.md` 与 Scheduling 相关模块文档需在首个实现 PR 中补引本 ADR；MES `RuleScheduler` 的 deprecated 状态写入 MES 模块文档。
6. 后续引入任何 solver、多目标优化、自动发布或仿真能力，必须新增 ADR 并重审 §9 的"计划员确认"门槛。
