# ADR 0002: 应用接入契约与 Agent/Connector 协议

- Status: Proposed
- Date: 2026-05-13

## Context

Nerv-IIP 的首要价值不是承载业务逻辑，而是稳定接入、发现、控制与观测真实应用实例。受管目标可能运行在 Docker、Windows Service 与自定义 HTTP 进程环境中，平台不能把接入方式绑定到单一部署形态。

平台需要区分应用目录事实、实例状态事实、执行动作编排与本地资源探测，不能把这些职责全部塞进同一个服务或脚本。任何启停、备份、恢复与日志拉取动作都需要审计、幂等与失败原因回传，否则平台无法形成可信运维闭环。契约如果不版本化，平台与 Agent 将在字段和能力演进过程中快速失配。

## Decision

1. 平台统一通过 Agent Host 接入受管环境，本地资源适配由 Connector 实现。
2. 首批至少规划 Docker Connector、Windows Service Connector、HTTP Connector。
3. 应用接入契约固定包含五类基础能力：注册、心跳、能力声明、健康检查、实例状态同步。
4. 平台首批受控执行动作固定为启动、停止、重启、拉取日志、触发备份、触发恢复。
5. 应用接入元数据至少包含应用标识、版本、部署目标、能力清单、节点信息、环境信息。
6. 心跳与状态同步必须分离：心跳只证明 Agent 或实例仍存活，状态同步负责更新实例事实。
7. 所有执行动作必须带幂等键、审计记录、执行结果与失败原因。
8. 高风险动作后续再叠加审批与人工确认。
9. 能力声明与动作契约必须版本化，平台按能力版本协商可执行动作与参数结构。

## Rationale

1. 用 Agent Host 统一接入层，可以把平台核心服务与目标环境的异构性隔离开。
2. Connector 模式能在不改变平台核心协议的前提下扩展新的宿主环境。
3. 将心跳与状态同步拆分，可以避免把存活证明和业务状态事实混为一个脆弱接口。
4. 把幂等、审计、失败回传作为协议硬约束，可以让运维动作具备可追溯性与可恢复性。
5. 对能力声明做版本化，可以降低平台与 Agent 升级不同步时的破坏性。

## Consequences

1. Agent 侧实现成本会上升，因为每类 Connector 都要遵守统一协议并回报结构化结果。
2. 平台需要维护能力版本兼容策略，而不是只靠最新版本覆盖所有节点。
3. 某些简单场景下，直接远程调用会比 Agent 协议看起来更省事，但长期会牺牲可观测性、一致性与审计闭环。
4. 日后扩展更多高风险动作时，协议层需要继续演进审批、授权与回滚信息。

## Implementation Notes

1. 首批脚手架需要创建 Agent Host 最小宿主、Docker Connector 原型以及注册、心跳、状态同步三条基本链路的契约定义。
2. AppHub 负责应用目录与实例事实，Ops 负责执行动作、任务记录与审计闭环，二者职责需要在接口设计中严格分开。
3. v1 协议基线、共享契约落点与首批公开接口以 docs/architecture/agent-platform-protocol-v1.md 为准；若后续调整传输机制，应在该文档中增量演进而不是回退到口头约定。

## Out of Scope

1. 不在本 ADR 中决定 Agent 与平台之间最终采用 HTTP 长连接、轮询还是消息驱动。
2. 不在本 ADR 中定义每个动作的完整参数模型与错误码表。
3. 不在本 ADR 中覆盖审批流、人工确认 UI 与回滚策略的最终实现细节。
