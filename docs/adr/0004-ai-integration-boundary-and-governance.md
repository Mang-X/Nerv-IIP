# ADR 0004: AI Integration 边界与治理模型

- Status: Proposed
- Date: 2026-05-13

## Context

Nerv-IIP 需要提供 AI 原生能力，但平台的核心价值在于管理与编排，不在于自建模型托管平台。AI 能力同时涉及外部模型提供方、MCP 工具、技能注册、权限校验、执行审批与人机协同，这些职责不能散落在各个后端服务中。

项目已经确认采用官方 .NET AI 生态，并优先使用 Microsoft.Extensions.AI 维持提供方可替换性。高风险动作如果直接由模型或工具执行，会带来明显的运维与审计风险，需要正式治理边界。Knowledge 服务已经被单独定义为文档引入与检索能力，不应与 AI 工具治理混杂在同一服务内。

## Decision

1. AI Integration 作为独立服务，负责模型提供方配置、MCP Server、Skill 注册、工具授权、执行审批与人机确认编排；工具授权必须基于 IAM 提供的身份、组织、环境、权限范围和外部客户端授权事实。
2. 平台不提供内置模型托管能力；模型统一通过外部 API 配置或外部 Agent 客户端接入。
3. 平台内部凡是需要直接调用模型的场景，默认通过 Microsoft.Extensions.AI 抽象接入。
4. AI Integration 不负责文档入库、分块、嵌入、向量索引与检索存储，这些职责由 Knowledge 服务承担。
5. 首批对外开放的工具以查询类能力为主；执行类工具仅开放低风险动作，并默认纳入审计与授权控制。
6. 高风险动作必须预留审批、人工确认或双重确认机制，不允许直接无审计执行。
7. Microsoft Agent Framework 不作为首期基础依赖，仅在确有长流程、多代理或复杂人机协同时再评估引入。

## Rationale

1. 将模型接入与工具治理集中到 AI Integration，并让工具授权消费 IAM 的身份与授权事实，可以避免各服务重复维护模型配置、权限与审计逻辑。
2. 使用 Microsoft.Extensions.AI 作为统一抽象，有利于保留模型提供方替换空间，降低供应商锁定。
3. 把 Knowledge 从 AI Integration 中分离，可以让知识管道与工具治理各自独立演进，减少服务职责膨胀。
4. 先以查询类工具起步，再逐步放开执行类动作，符合平台早期风险控制要求。
5. 将高风险动作纳入审批与确认，是工业互联网场景下的必要治理前提。

## Consequences

1. AI Integration 会成为模型调用与工具执行的关键治理点，需要较高的稳定性与审计能力。
2. 某些服务原本可以直接调用模型，现在需要经过统一抽象或治理边界，接入成本会略有增加。
3. 若后期引入更复杂的 agentic workflow，AI Integration 需要继续扩展状态管理与编排能力。
4. 平台需要维护工具权限模型、审批策略与模型配置生命周期，而不是只做简单 SDK 封装。

## Implementation Notes

1. 首批实现应覆盖模型配置、工具注册、工具授权、审计记录与人工确认挂点。
2. MCP 查询工具优先对接应用状态、实例详情、任务记录、日志检索与知识检索。
3. 若后续出现长流程、多代理或复杂人机协同，再评估是否引入 Microsoft Agent Framework 作为补充，而不是首期默认依赖。

## Out of Scope

1. 不在本 ADR 中决定具体模型提供方、模型型号与提示词模板细节。
2. 不在本 ADR 中定义审批流 UI、消息通知样式与人工确认交互细节。
3. 不在本 ADR 中决定 agentic workflow 的最终编排框架。
