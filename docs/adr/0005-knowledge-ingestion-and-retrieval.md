# ADR 0005: Knowledge 数据引入、索引与检索架构

- Status: Proposed
- Date: 2026-05-13

## Context

Nerv-IIP 的 AI 价值不仅来自工具调用，还来自对业务文档、运行手册、接入规范与历史知识的可检索化。文档引入、分块、嵌入、索引与权限过滤属于持续演进的数据管道，不应附属于前端、Gateway 或 AI Integration 的临时实现中。

项目已确认优先采用 Microsoft.Extensions.DataIngestion 与 Microsoft.Extensions.VectorData 作为 .NET 官方知识处理路径。知识检索结果不仅要供 AI 使用，也要支持平台 UI、运维排障与审计场景，因此需要独立服务边界。受管文档来源、权限模型与更新频率会持续变化，需要一个可扩展的 ingestion 架构而不是一次性脚本。

## Decision

1. Knowledge 作为独立服务，负责知识源接入、文档抽取、分块、嵌入、向量索引、检索、权限过滤与引用回显。
2. 文档处理主路径优先采用 Microsoft.Extensions.DataIngestion 与 Microsoft.Extensions.VectorData。
3. 向量索引默认落到 Qdrant；知识源原始文件与派生附件默认落到 MinIO 或等价对象存储。
4. 引入流程采用异步管道模式，允许部分失败重试、增量更新与重建索引。
5. 检索结果必须返回可追溯引用，不允许只返回脱离来源的纯文本片段。
6. 检索请求必须纳入组织、环境与权限上下文过滤，不允许绕过平台权限模型直接暴露原始知识内容。
7. Knowledge 对外提供统一检索能力；AI Integration、Gateway 与平台服务只消费其检索接口，不自行维护私有向量索引实现。

## Rationale

1. 将 Knowledge 独立出来，可以让知识引入与检索成为平台一级能力，而不是 AI 功能的附属实现。
2. 使用官方 .NET 数据引入与向量抽象，有利于保持技术路线一致并降低自造管道成本。
3. 异步 ingestion 更适合面对大批量文档、失败重试与持续更新场景。
4. 引用回显与权限过滤是工业场景可信检索的底线，没有这两点，知识能力很难用于真实运维与决策支持。
5. 统一检索服务可以避免多个系统各自维护一套分块、嵌入与索引逻辑。

## Consequences

1. Knowledge 会引入独立的数据处理与索引运维成本，需要关注任务积压、重建时间与存储增长。
2. 文档接入链路需要处理来源变化、格式异常与权限漂移，工程复杂度高于简单向量检索示例。
3. 平台各处若要消费知识能力，必须接受统一权限过滤与引用规则，不能为了便捷绕过服务边界。
4. 如果未来接入 OCR、结构化解析或多模态处理，Knowledge 服务的管道会继续扩展。

## Implementation Notes

1. 首批实现建议先支持文档来源注册、基础分块与嵌入、索引写入、查询检索与引用回显。
2. 知识源生命周期说明见 docs/architecture/knowledge-source-lifecycle.md，明确首次导入、增量更新、重建索引与删除失效文档的行为。
3. Knowledge 对外接口需要区分面向 UI 的检索响应与面向 AI/RAG 的检索响应，但底层权限与引用规则保持一致。

## Out of Scope

1. 不在本 ADR 中决定嵌入模型、rerank 策略与 chunk 参数的最终调优值。
2. 不在本 ADR 中决定 OCR、表格提取与多模态处理的具体供应商。
3. 不在本 ADR 中定义知识管理后台的最终交互界面与运营流程。
